# Three.js

> **Statut : composant de seconde génération**

Le composant **Three.js** est le premier composant développé selon les nouvelles conventions du framework Zealot.

Au-delà de l'intégration de Three.js, son implémentation a permis de définir une architecture commune pour les futurs composants JavaScript du CMS.

Cette documentation décrit cette architecture et les conventions retenues.

---

# Objectifs

Le composant poursuit plusieurs objectifs :

- intégrer des scènes 3D dans le CMS ;
- séparer complètement PHP et JavaScript ;
- rendre le composant extensible ;
- permettre plusieurs métiers (viewer, éditeur, atelier...) ;
- préparer les composants composites.

Le composant ne doit jamais contenir de logique métier spécifique.

Le métier est porté par les **Resources**.

---

```
Descriptor
    │
    ▼
ThreeRenderer
    │
    ▼
<div class="cp_threejs"
     data-options="{...}">
    │
    ▼
three/index.js
    │
    ▼
ComponentFactory
    │
    ▼
Viewer
    │
    ▼
ResourceLoader
    │
    ▼
ResourceRegistry
    │
    ▼

```

Cette architecture est volontairement modulaire.

Chaque niveau possède une responsabilité unique.

# Architecture PHP

Le Renderer ne contient aucune logique Three.js.

Il produit uniquement le conteneur HTML.

```
Descriptor
    │
    ▼
ThreeRenderer
    │
    ▼
<div
    class="cp_threejs"
    data-options="...">
```

Le Renderer ignore :

- les scènes ;
- les modèles ;
- les textures ;
- les lumières ;
- la caméra.

Ces informations appartiennent au JavaScript.

---

# Contrat Descriptor → JavaScript

Le Renderer transmet un unique objet :

```php
$options = $descriptor->get('options', []);
```

Le HTML devient :

```html
<div
    class="cp_threejs"
    data-options="{...}">
</div>
```

Le JavaScript lit uniquement :

```javascript
element.dataset.options
```

Aucun autre `data-*` n'est utilisé.

---

# Contrat options

Toutes les informations fonctionnelles sont regroupées dans `options`.

Exemple :

```json
{
    "type": "viewer",

    "resource": "cube",

    "background": "#202020",

    "resourceOptions": {

        "size": 1,

        "color": "#00AAFF",

        "rotationSpeed": 0.5
    }
}
```

Le Renderer n'a plus besoin d'évoluer lorsque de nouvelles fonctionnalités apparaissent.

---

# Viewer

Le Viewer encapsule complètement Three.js.

Responsabilités :

- créer la scène ;
- créer le renderer WebGL ;
- créer la caméra ;
- gérer le cycle de vie ;
- gérer le redimensionnement ;
- gérer la boucle d'animation ;
- charger les ressources.

Il ne contient aucune logique métier.

---

# ResourceLoader

Le Viewer délègue entièrement le chargement des ressources.

```
Viewer -> ResourceLoader 
            │
            ▼
        ResourceRegistry -> CubeResource
```

Le Viewer ignore la nature de la ressource chargée.

---

# ResourceRegistry

Chaque ressource est enregistrée une seule fois.

Exemple :

```javascript
ResourceRegistry.register( { resource: "cube" , component: CubeResource } );
```

Le registre prépare l'arrivée de ressources plus complexes :
- GLTF
- Terrain
- Train
- Usine
- Jumeau numérique

---

# Resource

Une Resource représente un objet métier.

Exemple :

```
CubeResource
```

Une Resource :

- construit ses objets Three.js ;
- ajoute les objets à la scène ;
- anime ses objets ;
- détruit proprement les ressources.

Une Resource ne crée jamais :

- la caméra ;
- le renderer ;
- la scène ;
- le canvas.

Ces responsabilités appartiennent au Viewer.

---

# Cycle de vie

Le framework distingue deux chemins d'exécution.
1. Rendu initial (PHP → JavaScript)


## Rendu initial (PHP → JavaScript)
Le composant est rendu lors du chargement d'un article.
```
Descriptor -> DescriptorMapper -> Renderer
                                    │
                                    ▼
                        <div class="cp_xxx"
                             data-options="...">
                                    │
                                    ▼
                                Vue CMS ->    bootstrap.js -> index.js
                                                                │
                                                                ▼
                                                            ComponentFactory
                                                                │
                                                                ▼
                                                            Viewer
```
C'est le chemin utilisé aujourd'hui par :

Apex
Mermaid
Leaflet
Three.js
Callout
CodeVal
##  Rendu dynamique (EventBus)

Le composant existe déjà.  Une action déclenche son évolution.
```
EventBus -> Viewer
            │
            ▼
        refresh()
        render()
        resize()
        destroy()
```
## Note
- Le bootstrap ne participe jamais aux événements. Il n'intervient qu'une seule fois.
- Par convnetion tous les composants exposent la même interface.

## API uniforme.

Tous les composants exposent la même interface.
Certaines méthodes peuvent être vides selon le composant.
```
init()
refresh()
resize()
render()
animate()
destroy()
```

---

# ComponentFactory

Le Factory instancie les composants.

```
ComponentFactory 
    │
    ▼
Viewer
```

Le Factory ne connaît jamais les ressources.

---

# ComponentRegistry

Le registre associe un type de composant à son implémentation.

Exemple :

```javascript
ComponentRegistry.register( { type: "viewer" , component: Viewer } );
```

Cette architecture prépare :

- Viewer
- Editor
- Workspace

sans modifier le reste du framework.

---

# bootstrap

Le bootstrap reste volontairement minimal.

```
bootstrap.js

↓

three.init(document)
```

Toute la logique est portée par le composant.

---

# AdminRenderer

Le rôle du AdminRenderer est uniquement d'éditer le Descriptor.

Il ne produit jamais le composant.

Le formulaire manipule progressivement :

```
config[id]

config[options][...]
```

Cette structure deviendra le contrat commun du framework.

---

# Composants composites
Demain, un composant composite deviendra lui-même un nœud.
Chaque enfant conserve son propre cycle de vie.
Le composant composite orchestre uniquement les échanges.
```
bootstrap
    ↓
CompositeComponent
    ├── Viewer
    ├── Apex
    ├── CodeVal
    └── Mermaid
```
## Exemple :
```
Atelier moteur
├── CodeVal
├── Apex
└── Three.js
```

# Conventions appliquées

Le composant Three.js applique les conventions suivantes :

| Convention | Objet |
|------------|-------|
| C001 | Préfixe `cp_` pour tous les composants |
| C002 | Renderer sans logique métier |
| C003 | Communication PHP → JS via `data-options` |
| C004 | Séparation composant / ressources |
| C005 | Registry + Factory |
| C006 | AdminRenderer dédié au Descriptor |
| C007 | API publique indépendante de l'implémentation |
| C008 | Préparation des Node et CompositeComponent |
| C009 | Les événements appartiennent aux composants |
| C010 | APIs utilisant des objets de contexte |
| C011 | Responsabilité unique des classes |
| C012 | Cycle de vie uniforme |
| C013 | Registry à chaque niveau du framework |
| C014 | Développement par incréments vérifiables |
| C015 | API homogène pour tous les Registry |
| C016 | Imports absolus depuis `/assets/js/...` |

# Retrofit

Three.js est le premier composant développé selon l'architecture de seconde génération.

Les composants existants (Apex, Leaflet, Mermaid, CodeVal, Callout) migreront progressivement vers :

- le contrat `options` ;
- le cycle de vie uniforme ;
- les Registry ;
- les Factory ;
- les nouveaux AdminRenderer.

La compatibilité avec les composants existants reste conservée pendant la période de transition.

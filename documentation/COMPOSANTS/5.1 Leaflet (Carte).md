# Leaflet

Leaflet est le premier composant qui cumule presque toutes les difficultés :

ressources CSS externes ;
bibliothèques JS externes ;
plugin IGN ;
fichier customConfig.json ;
initialisation asynchrone ;
problème de visibilité (display:none) ;
plusieurs couches ;
état interne (map, layers, markers, etc.).

Si on arrive à intégrer Leaflet proprement sans casser l'architecture, c'est que les conventions sont bonnes.

##### app/Libraries/Components/Renderers/LeafletRenderer.php

utiliser des data-* plutôt qu'un JSON complet pour cette première itération :
ils sont lisibles directement dans le DOM ;
faciles à déboguer avec les outils du navigateur ;
simples à récupérer en JavaScript (el.dataset.lat, etc.) ;
cohérents avec ce que fait déjà ApexRenderer.

Le jour où Leaflet devra gérer des objets plus complexes (liste de couches, marqueurs, styles, GeoJSON...), on pourra passer à un unique attribut :
```
data-config='{"lat":47.82,"lng":-4.30,"layers":["osm"]}'
```

---

##### app/Libraries/Components/AdminRenderers/LeafletAdminRenderer.php

rester sur la même philosophie qu'ApexAdminRenderer : 

un formulaire minimal mais complet. Le but est uniquement de pouvoir configurer une carte OSM.

quatre champs correspondent exactement au LeafletRenderer :
config[id]
config[lat]
config[lng]
config[zoom]

On évite d'introduire dès maintenant :
- les couches ;
les marqueurs ;
les fonds de carte ;
les plugins IGN ;
les options Leaflet.
Une petite amélioration

**profiter de Leaflet pour améliorer tous les futurs AdminRenderer**.

Plutôt que :
```
$lat = $descriptor->get('lat', 47.82);
```
je définirais des constantes dans la classe :
```
protected const DEFAULT_LAT  = 47.82;
protected const DEFAULT_LNG  = -4.30;
protected const DEFAULT_ZOOM = 11;
```
Puis :
```
$lat  = $descriptor->get('lat', self::DEFAULT_LAT);
$lng  = $descriptor->get('lng', self::DEFAULT_LNG);
$zoom = $descriptor->get('zoom', self::DEFAULT_ZOOM);
```
Ainsi, les valeurs par défaut sont centralisées et faciles à modifier (par exemple si tu souhaites centrer par défaut les cartes sur une autre région). Nous pourrons généraliser cette pratique aux autres composants lors de leur refactorisation.
